language, (C) ratify the deviation as a sanctioned exception. Operator

(4) Compass alignment attempt revealed unexpected state. Aggressive
override applied to compass.html — `#compass-app { --bg1: #040f23cf;
--bg2: #040f23cf }` at higher specificity than hud.css's :root. Console
diagnostics confirmed the override propagated to every consuming
surface in compass.html. Visual review showed no change. Sequence of
diagnostic snippets walked through specificity, deployment cache,
DOM-level computed values, and finally an inline-painted sentinel
that confirmed the renderer was responsive. Conclusion: Compass had
already been rendering at the target color before the override; the
override was a no-op cosmetically. The original premise — "Compass is
darker than dashboard, align dashboard to it" — was correct. Dashboard
was the outlier; dashboard.html's #040f23cf override brought it into
alignment with Compass. The Compass override remained in place to
guarantee the parity didn't drift.

(5) But one Compass panel remained visually distinct — Recommended
Sequence. Console diagnostics on its outer container showed
background:rgba(0,210,255,.03), not a token reference. Walked the
DOM up to find the rendering origin: not in compass.html or hud.css,
but in mw-sequence.js line 299, an inline style:"" attribute on the
panel container's <div>. The architectural smell crystallized at this
point — panel chrome was being rendered from a behavior file with
hardcoded inline styles, unreachable to any token override at any
specificity. The pattern is exactly what doctrine §6.2 (introduced
v1.2) already forbids; the rule had been correct since v1.2 but had
never been audited against Compass's JS-rendered surfaces.

Architect operating-practice lessons logged this session
(continuing the catalog from the prior journal entries):

(22) Operator-as-acting-architect single-session ratification is a
sanctioned bridge mode when the architect role is unstaffed. The
authority chain in §0.1 names operator (product) + architect (code) as
the ratification dyad. When only the operator is present, the operator
may stand in as acting-architect for a single session, ratify the
doctrine amendment, and proceed. The status line on the doctrine
records the substitution explicitly ("ratified by operator and
acting-architect (operator standing in for architect role per
2026-04-28 session)"). Distinguishes from the v1.1 incident where the
coding agent inadvertently played the architect role — that was a
bridge-corruption failure (lesson 18). Operator-as-acting-architect is
a bridge-shortening mode under operator control. Limit: the operator
should not stand in as acting-architect for changes the operator
cannot evaluate technically (e.g., pure code-correctness amendments
about thread-safety, schema, performance). The v1.7 amendments were
all visual / structural / discipline — operator could evaluate them
fully.

(23) Console diagnostic snippets are the primary instrument for
surfacing §6.2 violations on JS-rendered surfaces. The Compass audit
relied on a single querySelectorAll filter snippet to inventory all
inline-styled background elements; classification by inspection was
straightforward once the inventory was in hand. Future doctrine
applications on JS-rendered surfaces (Cadence, Aegis, /scripts/*)
should use the same instrument. The snippet is preserved in CMD-A's
§1; future briefs that touch JS-rendered surfaces should reference it
or copy it. Distinguishes a class of doctrine-application work that
cannot be discovered by reading source files alone — only the
rendered DOM contains the truth.

(24) Layer 1 token override at a high-specificity wrapper
(`#compass-app { --bg1: #040f23cf; --bg2: #040f23cf }`) is currently
unaddressed in doctrine. The pattern preserves the Layer 2 semantic
alias system (--surface-* aliases continue to point at --bg1/--bg2),
preserves the elevation ladder structure, but allows a surface to
locally retune the absolute color values that the ladder resolves to.
It is structurally different from collapsing tier separation
(dashboard's two-class override that flattened panel + KPI to a
single fill — that was a doctrine violation and the original v1.7
question). It is also structurally different from §3.5.6's surface-
local panel-background overrides that v1.6.1 retired (those targeted
panel-class backgrounds directly, not the underlying tokens). The
ProjectHUD doctrine has no language about whether token-value local
retunes are sanctioned, forbidden, or sanctioned-with-conditions.
Should be addressed in v1.8 or as a v1.7.1 point-release. Three
candidate dispositions: (a) sanction explicitly with a "scoped
retune" pattern that records the surface and the operator-ratified
values; (b) forbid explicitly — token values are global and
amendments to them require doctrine ratification, not surface-local
overrides; (c) defer with a watchful-waiting clause until the pattern
recurs or causes a regression. Logged for next architect review.

(25) Visual-parity exercises across surfaces benefit from running
console diagnostics on both surfaces in parallel before committing to
a direction. The Compass-vs-dashboard parity work consumed several
diagnostic round-trips before reaching the conclusion that Compass
was already at the target — a single side-by-side console diagnostic
("compute background of canonical panel on each surface") at the start
would have surfaced this in one pass. Future visual-parity work
should default to "diagnose both surfaces, compare, then decide
direction" rather than "edit one surface and review."
